# Notifications: push delivery + Munshi AI suggestions — design

Status: approved (design), pending implementation plan
Date: 2026-07-06

## 1. Problem

Today "notifications" are in-app only: DB rows rendered on the Notifications
screen and a bell dot on Home. There is **no creation logic** (rows exist only
via the seed) and **no OS push** — nothing reaches a physical phone, and no
event ever generates a notification.

Goal: make notifications fully functional end-to-end —

- Real OS push to a physical phone (lock screen / notification shade) **and**
  the existing in-app list.
- Notifications actually generated by app events and schedules.
- An "interesting" AI angle: a Munshi (bookkeeper persona) suggestion that only
  fires when there is genuinely something worth saying.

## 2. Scope

In scope (v1):

- Trigger types: `large_transaction`, `budget_alert`, `goal_progress`,
  `monthly_report`, and a new `munshi_suggestion`. (`security_alert` remains an
  enum value but has no automated trigger in v1.)
- Expo push registration + delivery.
- Tap → deep-link to the relevant screen.
- Settings toggles wired to backend preferences.

Out of scope (v1):

- Security/login-detection notifications (no trigger).
- Per-user quiet hours / custom delivery time (fixed daily time, config-level).
- Rich/actionable notification buttons.
- Read-state sync pushed back to the device badge count.

## 3. Architecture

Three cooperating layers. Real-time events and scheduled jobs both funnel into a
single creation choke-point that writes the in-app row and dispatches push.

```
EVENTS (real-time)            SCHEDULE (cron, @nestjs/schedule)
transaction.created ─┐        daily 09:00 IST ─┐
goal.contribution  ─┤        monthly 1st ──────┤
                     ▼                          ▼
        NotificationsListener          MunshiSuggestion / MonthlyReport
        (rule evaluation)              generators
                     │                          │
                     └───────────┬──────────────┘
                                 ▼
                    NotificationsService.create()
                    • saves in-app row (history)
                    • PushDispatcher → Expo Push API
                                 ▼
                      Physical phone (banner / shade)
                                 ▼ tap
                    Mobile deep-link → relevant screen
```

**Event-driven triggers (key isolation choice).** `TransactionsService` and
`GoalsService` do NOT import `NotificationsService`. They emit events via NestJS
`EventEmitter2` (`@nestjs/event-emitter`). A `NotificationsListener` inside the
notifications module subscribes and evaluates rules. This keeps notification
logic fully isolated, avoids circular module dependencies, and leaves the
transaction/goal write paths unaware of notifications.

## 4. Backend components

### 4.1 New / changed data

- **`DeviceToken` entity** (new): `id`, `userId`, `expoPushToken` (unique),
  `platform` ('ios' | 'android'), `createdAt`, `lastSeenAt`. A user may have
  multiple devices.
- **`NotificationType` enum** (changed): add `MUNSHI_SUGGESTION = 'munshi_suggestion'`.
- **`UserPreferences`** (changed): add `munshiSuggestionsEnabled` (default true)
  and `monthlyReportEnabled` (default true). Existing `notificationsEnabled`
  (master), `budgetAlertsEnabled`, `goalMilestonesEnabled`, `largeTxAlertsEnabled`
  are reused. `update-preferences.dto` extended accordingly.
- **`Notification.data`** (new column): `jsonb`, nullable — carries deep-link
  payload `{ screen, id? }`.

### 4.2 Services

- **`NotificationsService.create(userId, { type, title, body, data })`** — the
  single creation choke-point. Applies gating (§7), writes the in-app row, then
  (if allowed) calls `PushDispatcher.send`.
- **`PushDispatcher`** (new service) — loads the user's `DeviceToken`s, builds
  Expo push messages (`to`, `title`, `body`, `data`, Android channel), POSTs to
  `https://exp.host/--/api/v2/push/send` in chunks, and prunes tokens whose
  receipt/response is `DeviceNotRegistered`. Network/HTTP failures are logged and
  swallowed — a push failure must never break the originating request or job.
- **`NotificationsListener`** (new) — `@OnEvent('transaction.created')`,
  `@OnEvent('goal.contribution')`. Runs rule evaluators, calls `create()`.
- **`MunshiSuggestionService`** (new) — daily generation (§6).
- **`MonthlyReportService`** (new, or a method on reports) — monthly generation.

### 4.3 Endpoints (notifications controller)

- `POST /notifications/register-device` — body `{ expoPushToken, platform }`.
  Upserts the token for the current user, updates `lastSeenAt`.
- `POST /notifications/unregister-device` — body `{ expoPushToken }`.
- Existing `GET /notifications`, `POST /notifications/read-all`,
  `POST /notifications/:id/read` unchanged.

### 4.4 Module wiring

- `NotificationsModule` imports what the listener/generators need
  (Budgets, Goals, Transactions, Reports services; the shared Anthropic client
  provider) without those modules importing it back.
- `EventEmitterModule.forRoot()` registered globally in `AppModule`.
- `ScheduleModule.forRoot()` registered.

## 5. Trigger rules

| Type | Fires when | Dedup | Deep-link `data` |
|---|---|---|---|
| `large_transaction` | new **expense** with amount ≥ `LARGE_TX_THRESHOLD` (config, default ₹20,000) | one per transaction | `{ screen:'tx-detail', id: txId }` |
| `budget_alert` | an affected category **or** the budget total *crosses* 90% or 100% of cap due to this tx | once per budget per threshold per period (query recent rows to confirm it's a fresh crossing) | `{ screen:'budgets' }` |
| `goal_progress` | goal crosses a 25 / 50 / 75 / 100% bucket on a contribution | once per bucket per goal | `{ screen:'goals' }` |
| `munshi_suggestion` | daily cron, only if noteworthy (§6) | max 1/day/user | `{ screen:'chat' }` |
| `monthly_report` | cron on the 1st, 09:00 IST | 1/month/user | `{ screen:'reports' }` |

**Crossing semantics:** "crosses" means the metric was below the threshold
before this event and is at/above it after — not merely "is above". Dedup for
budget/goal is enforced by checking for an existing notification of that type for
that budget/goal within the current period/bucket before creating.

Cron schedules and `LARGE_TX_THRESHOLD` come from config (env), defaulting to
09:00 Asia/Kolkata and ₹20,000.

## 6. Munshi AI suggestion — "only when noteworthy"

`MunshiSuggestionService`, run daily per opted-in user:

1. **Cheap pre-filter (no LLM):** skip entirely when there is nothing to react to
   — no recent activity, nothing near a budget cap, no goal near a milestone.
   This keeps cost and noise down.
2. **Generate:** build the same budget / active-goals / recent-transactions
   snapshot the chat uses (`buildPromptContext`-style), and make **one
   non-streaming `messages.create`** call (small `max_tokens`) with a compact
   prompt that reuses Munshi's persona and MUST return strict JSON — either
   `{"skip": true}` or `{"title": "...", "body": "..."}`. The model is instructed
   to skip on a boring day.
3. **Emit:** on a non-skip result, `NotificationsService.create({ type:
   munshi_suggestion, title, body, data:{ screen:'chat' } })`.

Guardrails: parse defensively (treat unparseable / malformed output as skip);
clamp title/body length; never fabricate numbers not in the snapshot (persona
prompt already forbids this). Model id comes from the existing `AI_MODEL` config
(default `claude-opus-4-8`); if the Anthropic client is absent (no key), skip.

## 7. Gating

- **Per-type toggle off** (`budgetAlertsEnabled`, `goalMilestonesEnabled`,
  `largeTxAlertsEnabled`, `munshiSuggestionsEnabled`, `monthlyReportEnabled`) →
  that category is not created at all.
- **Master `notificationsEnabled` off** → the in-app row is still written (history
  stays useful) but the phone push is suppressed.
- **No registered device token / permission denied** → in-app rows still written;
  only phone delivery is skipped.

## 8. Mobile

Stack: managed Expo (v56) + EAS. Read the versioned Expo v56 docs before writing
notification code (per `mobile/AGENTS.md`).

- **Dependencies:** `expo-notifications`, `expo-device`. Add the
  `expo-notifications` plugin + Android notification channel config to `app.json`.
- **Registration:** after login (where a valid JWT exists), request notification
  permission; on grant, `getExpoPushTokenAsync()` → `POST
  /notifications/register-device`. On Android, create the default notification
  channel. Physical Android requires a dev/preview build (Expo Go dropped remote
  push on Android in SDK 53+).
- **Foreground handler:** `setNotificationHandler` + a received-listener that
  refreshes the in-app list and bell dot when a push lands with the app open.
- **Tap handler (deep-link):** `addNotificationResponseReceivedListener` reads
  `data.screen` (+ optional `data.id`) and calls `nav()` on the existing
  `navContext`. A pure `mapNotificationToScreen(data)` function does the mapping
  and is unit-tested. Handles the cold-start case (app launched from a
  notification) via `getLastNotificationResponseAsync()`.
- **Notifications screen / adapter:** add the `munshi` view type
  (`NOTIF_TYPE_MAP`, `NOTIF_COLORS`, `NOTIF_ICONS` — icon 🧮, its own color) and a
  "Munshi" filter chip.
- **Settings:** wire master + budget/goal/large-tx toggles to backend prefs; add
  Munshi + Monthly-report toggles. Toggling the master toggle also
  registers/unregisters the device where appropriate.

## 9. Data flow (concrete example — large transaction)

1. User adds a ₹28,000 rent expense → `TransactionsService.create` saves it and
   emits `transaction.created` with the tx.
2. `NotificationsListener` sees amount ≥ ₹20,000, checks `largeTxAlertsEnabled`.
3. `NotificationsService.create({ type: large_transaction, title:'Large
   transaction detected', body:'₹28,000 debited — Rent.', data:{screen:'tx-detail',
   id} })` → row saved.
4. Master toggle on + token present → `PushDispatcher` POSTs to Expo → phone shows
   a banner.
5. Tap → app opens → `mapNotificationToScreen({screen:'tx-detail',id})` →
   `nav('tx-detail', {id})`.

## 10. Testing

Backend (jest, matching existing `*.spec.ts`):

- Rule evaluators: crossing math (below→at/above), large-tx threshold boundary,
  goal milestone bucket transitions, per-period dedup.
- Munshi generation: pre-filter skip, JSON parse (skip / emit / malformed→skip),
  length clamp; Anthropic client mocked.
- `PushDispatcher`: payload/chunk building, `DeviceNotRegistered` pruning,
  failure-swallowing; `fetch` mocked.
- Gating matrix: per-type off, master off (row written, no push), no token.

Mobile:

- Pure `mapNotificationToScreen` mapping for each type.
- Token-registration function (permission granted / denied paths) with
  expo-notifications mocked.

## 11. One-time setup (ops, not code)

- EAS credentials: FCM (Android) and APNs (iOS) configured once so Expo push can
  deliver.
- Physical Android testing requires a preview/dev build, not Expo Go.
- New config/env: `LARGE_TX_THRESHOLD`, cron times (default 09:00 Asia/Kolkata).
  `AI_MODEL` already exists.

## 12. Implementation phasing

1. Backend: `DeviceToken`, `NotificationsService.create`, `PushDispatcher`,
   register/unregister endpoints, event emission from transactions/goals,
   `NotificationsListener` + real-time rules, gating, preference fields.
2. Backend: `ScheduleModule`, `MunshiSuggestionService` (daily), monthly report
   job.
3. Mobile: expo-notifications setup, registration, foreground + tap/deep-link
   handlers, adapter/screen `munshi` type, Settings toggles.
